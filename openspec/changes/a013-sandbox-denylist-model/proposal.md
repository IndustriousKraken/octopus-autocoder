## Why

`a006` masks the entire home directory and binds back only the workspace + the role's own CLI store. On a real dev server that breaks the executor: build toolchains install under `$HOME` (`~/.cargo`/`~/.rustup` for Rust, `~/.nvm`, `~/.pyenv`, `~/.rbenv`, …), and masking home hides all of them — so the executor can't run the very `cargo test` / `npm test` its tasks require. Whitelisting every possible toolchain is unbounded and brittle (version-pinned PATH entries, off-PATH shims), and a sandbox that breaks builds gets `allow_unsandboxed: true`'d — at which point nothing is protected.

The correct posture for this threat model (an automated dev server whose only secrets are dev credentials) is the inverse: **expose the home directory and mask a bounded, known set of sensitive paths.** The set that must be hidden — ssh, cloud tokens, gpg, the other CLIs' stores, package-manager credential files, and shell-init/persistence files — is small and slow-changing; the set of tools is not. A working denylist that stays enabled protects more than a fail-closed allowlist that gets switched off. "Secure by default" means the *dangerous* things are closed, not that everything required to work is.

This change flips the executor's filesystem model to a denylist, keeps the strict allowlist for read-only roles (and as an opt-in compliance mode), and folds in the macOS provider (superseding `a73`) and the CLI-binary binding (superseding `a012`, whose concern survives for the allowlist roles).

## What Changes

**The filesystem policy becomes role-dependent.** The **executor** runs under a denylist: `$HOME` is present and writable (toolchains and caches just work, no enumeration), EXCEPT a default-deny **mask-list** of sensitive paths. The mask-list covers two categories — **credential paths** (read-protection: `~/.ssh`, `~/.aws`, `~/.gnupg`, `~/.netrc`, cloud-token dirs, other CLIs' stores, `~/.cargo/credentials.toml`, `~/.npmrc`, `~/.pypirc`, `~/.gem/credentials`) and **shell-init/persistence paths** (write-protection: `~/.bashrc`, `~/.profile`, `~/.ssh/authorized_keys`, autostart/cron). It ships with defaults and is operator-editable (add a path to mask, or remove a default to expose it — e.g. open `~/.ssh` to develop an SSH tool — explicit and logged).

**Read-only roles (audits, agentic reviewer) keep the allowlist:** home masked; they see only the read-only workspace, their own CLI store, the **resolved CLI binary + its dependency closure** (following symlinks, even under `~/.local/bin` — the folded `a012` fix), and the minimal runtime. They run no build, so they need no toolchain. **Strict mode** is the same allowlist offered as an opt-in for the executor on high-compliance hosts; it is not the default.

**macOS folds in** (from `a73`): `sandbox-exec` with a generated Seatbelt profile realizes the same policy (exposed-home-minus-mask for the executor; allowlist for read-only roles), and the mechanism gate generalizes to "no platform-appropriate mechanism."

## Impact

- **Affected specs:** `executor` — MODIFY `Every agentic subprocess runs inside an OS-level sandbox` (role-dependent FS policy: executor denylist / read-only-role + strict-mode allowlist; macOS; CLI-binary binding) AND `CLI config stores are protected by OS-hide and engine-deny` (reframe `os_hide` as mask-list membership). `orchestrator-cli` — MODIFY `Sandbox credential-protection config — toggles, precedence, and relaxed-posture logging` (editable mask-list, strict-mode opt-in, macOS gate).
- **Affected code:** `sandbox.rs` — for the executor, `--ro-bind / /` + bind `$HOME` rw + `--tmpfs`/inaccessible over each mask-list entry (replacing `--tmpfs <home>`); for read-only roles + strict mode, the existing mask-home allowlist plus the resolved-CLI-binary bind (`which` + follow symlinks + dependency closure); the macOS `sandbox-exec` profile generator; the platform mechanism gate generalized. Config — the mask-list default set + per-repo/global editing + strict-mode flag.
- **Operator-visible behavior:** the executor's builds find the project toolchain with no per-tool configuration; `~/.ssh` and the other sensitive paths are masked by default (opening one is explicit + logged); read-only roles stay tightly home-masked. No `allow_unsandboxed` required for a normal dev box. Supersedes `a012` and `a73`.
- **Dependencies:** amends the canonical `a006`. Complemented by the env-capture / toolchain-activation change (sibling) — files visible here, commands made runnable there.
- **Acceptance:** `cargo test` passes; `cargo clippy --all-targets -- -D warnings` is clean; `openspec validate a013-sandbox-denylist-model --strict` passes. Tests: the executor reads `~/.cargo`/`~/.pyenv` and writes a cache while `~/.ssh`/`~/.cargo/credentials.toml` are masked; a masked persistence file write does not persist; a read-only role sees a home-masked allowlist with its CLI binary bound (under `~/.local/bin`); removing a default mask entry emits a startup relaxed-posture WARN; the macOS gate is satisfied via `sandbox-exec`.
