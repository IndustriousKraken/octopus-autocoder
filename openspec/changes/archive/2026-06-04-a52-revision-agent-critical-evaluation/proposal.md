## Why

A reviewer- or operator-triggered revision request can be wrong: it may be mistaken about the code, ask for a change that breaks a passing or spec-traced test, reference a symbol that does not exist, or churn working idiomatic code for protection that does not apply. The revision agent currently has no instruction to push back — it tends to apply the request as stated. Concrete reviewers tested have produced revision requests that would damage the codebase if applied verbatim (removing a spec-traced test believed redundant; rewriting working code for a non-applicable concern).

The revision prompt should direct the agent to evaluate the request critically: read the cited code, verify the claim against current state, and decline or partially honor the request when the claim is wrong — reporting what it declined and why through the `final_answer` summary that a45 surfaces in the revision comment. a45 plants the seed that "declining is a valid, reportable outcome"; this change adds the *how-to-evaluate* guidance on top of it.

**A latent bug blocks this from working end-to-end.** When the agent declines and makes no code change, the revision dispatcher's `Completed` path mishandles it. `apply_revision_commit` (`autocoder/src/revisions.rs:1210-1213`) unconditionally runs `git add_all` → `commit` → `push_force_with_lease`; on a clean working tree the commit fails, and the dispatcher's error branch posts `✗ Revision attempt failed: commit/push failed` — discarding the agent's declination reasoning and flagging a deliberate, correct decision as a failure. So encouraging declination without fixing this path would route every reasoned no-change outcome to a false failure. This change makes a clean-tree `Completed` a first-class reported declination.

## What Changes

**Revision prompt instructs critical evaluation (executor).** A new executor requirement states `prompts/implementer-revision.md` SHALL instruct the agent to verify the request's claim against the cited code before applying it, and to decline or partially honor a wrong request while reporting the decision via `final_answer`. This is design intent for the prompt's content, verified by review AND the drift audit — not a wording test (per the project-documentation requirement `Tests assert behavior or derivation, never message wording`). The guidance is written language-neutrally (it references "the project's test and lint commands," not a specific toolchain).

**No-change declination is a reported success, not a failure (orchestrator-cli).** The `Revision execution updates the agent branch and posts a reply comment` requirement is MODIFIED so the `Completed` outcome branches on the working-tree state: a dirty tree commits + force-pushes as before (and carries `final_answer` per a45); a clean tree is treated as a deliberate no-change declination — the daemon SHALL NOT attempt a commit (a clean tree is not a commit/push failure) and SHALL NOT post a failure comment, but posts a success comment whose first line marks an evaluation with no change made (distinct from `✅ Revision applied:`) followed by the agent's `final_answer`. Both branches count the attempt against the revision cap and advance the seen-marker.

**Stacks on a45.** a52 MODIFIES the same `Revision execution…` requirement a45 MODIFIES, building on a45's `final_answer` surfacing; a52's delta reproduces a45's version and extends it. a52's prompt edit builds on a45's outcome-signal section in `prompts/implementer-revision.md`. a45 (lower number) lands first.

## Impact

- **Affected specs:**
  - `executor` — ADDED `Revision prompt instructs critical evaluation of the reviewer's request` (prompt design intent; drift-audited; no content test).
  - `orchestrator-cli` — MODIFIED `Revision execution updates the agent branch and posts a reply comment` (stacking on a45): clean-tree `Completed` is a reported declination, not a failure. a45's `final_answer` scenarios and the `AskUser`/`Failed`/oversize-truncation scenarios are preserved.
- **Affected code:**
  - `prompts/implementer-revision.md` — extend the outcome-signal section (added by a45) with critical-evaluation guidance: read the cited code, verify the claim, decline or partially honor when wrong, report via `final_answer`. Language-neutral phrasing.
  - `autocoder/src/revisions.rs` — in the `Completed` arm (~line 943), check whether the agent produced changes (dirty working tree) before committing. Dirty → existing `apply_revision_commit` path. Clean → post the no-change declination success comment with `final_answer`; skip commit/push; increment the cap counter; advance the seen-marker. Distinguish from the genuine commit/push-failure path.
- **Operator-visible behavior:** when the revision agent declines a wrong request, the operator sees a success comment explaining what was declined and why, instead of a misleading `✗ Revision attempt failed`. Applied (dirty-tree) revisions are unchanged from a45.
- **Acceptance:** `cargo test` passes; `openspec validate a52-revision-agent-critical-evaluation --strict` passes. Tests: a `Completed` outcome with a clean working tree posts a no-change success comment carrying `final_answer`, makes no commit, and increments the cap; a `Completed` outcome with a dirty tree commits + pushes + posts `✅ Revision applied:` (a45 behavior); a genuine commit/push failure on the dirty path still posts the failure comment.
- **Dependencies:** stacks on **a45** (shared `Revision execution…` requirement + shared prompt file). Independent of a44/a46/a47/a51, though a51 and a52 both edit `prompts/implementer-revision.md` and should be applied after a45.
