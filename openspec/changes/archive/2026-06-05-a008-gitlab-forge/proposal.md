## Why

`a007` put the `Forge` trait in place with `GithubForge` as the sole, behavior-identical provider. This change is **Phase 2**: add `GitlabForge` so a GitLab-hosted repository is first-class for the daily loop ‚Äî autonomous merge-request creation, the reviewer, the revision loop, and authorized `@<bot>` triggers ‚Äî instead of just a pushed branch the operator turns into an MR by hand. The same per-repo `forge:` block carries an `api_base`, which makes **GitHub Enterprise** work for free (the GitHub shape against a self-hosted endpoint). Fork mode and the issues-lane ingestion side stay in Phase 3.

## What Changes

**`GitlabForge` implements the `Forge` trait against GitLab's API.** Repository parsing extracts the GitLab host and the URL-encoded `namespace/project` path (nested groups supported). MR lifecycle: create, list-open, find-by-source-branch, and draft toggling via GitLab's `Draft:` title prefix. Comments via MR notes. Reviews map the verdict to GitLab ‚Äî approve ‚Üí MR approval; request-changes and comment ‚Üí an MR note (GitLab has no distinct request-changes state). Authorization maps the commenter's GitLab member access level ‚Äî **Developer and above** (Developer / Maintainer / Owner) is authorized; Reporter and Guest are not ‚Äî mirroring the GitHub `author_association` gate. The push-only hint becomes GitLab-specific (`glab mr create` / MR web URL).

**An explicit per-repo `forge:` block selects and configures the provider.** Fields: `kind` (`github` | `gitlab`), `host`, optional `api_base`, and the token route. When present it is authoritative for provider selection; when absent, the provider defaults to GitHub against `github.com` (existing GitHub configs are unchanged and need no block). GitLab is selected ONLY via an explicit `forge: { kind: gitlab }` ‚Äî there is no host-sniffing fallback. `api_base` additionally enables GitHub Enterprise (`kind: github` against a self-hosted host/api_base).

## Impact

- **Affected specs:** `git-workflow-manager` ‚Äî ADD `GitLab forge provider`. `orchestrator-cli` ‚Äî ADD `Per-repo forge config block`.
- **Affected code:** `GitlabForge` in the forge module (MR / notes / approvals REST, GitLab repo parsing with nested groups, `Draft:` prefix toggling, access-level authorization, `branch_url`); config (`forge:` block parse + the selection precedence: explicit block ‚Üí `github.com` default ‚Üí no-provider error); GitLab token sourced from the forge block's token route, integrated with the existing token routing; the push-only branch hint.
- **Operator-visible behavior:** a repository with `forge: { kind: gitlab, host, token }` gets autonomous MRs, the reviewer, the revision loop, and authorized `@<bot>` triggers on GitLab; GitHub Enterprise via `forge: { kind: github, host, api_base }`; `github.com` repositories are unchanged and need no `forge:` block. A non-`github.com` host with no `forge:` block returns the clear no-provider error from `a007`, now directing the operator to declare a `forge:` block.
- **Dependencies:** stacks on `a007` (the `Forge` trait + `GithubForge`). Fork mode + issues ingestion are Phase 3. The archived `a000` authorization gate is now expressed through the trait ‚Äî `GithubForge::authorize` (author_association) and `GitlabForge::authorize` (access level).
- **Acceptance:** `cargo test` passes; `cargo clippy --all-targets -- -D warnings` is clean; `openspec validate a008-gitlab-forge --strict` passes. Tests: a `forge: { kind: gitlab }` config selects `GitlabForge`; MR create/list/find-by-source-branch round-trips against recorded fixtures; `set_pr_draft` toggles the `Draft:` prefix; `post_review` approve ‚Üí approval and request-changes/comment ‚Üí note; access level Developer+ is authorized and Reporter/Guest denied; a `github.com` repo with no `forge:` block still selects `GithubForge`; a GitLab-host URL with no `forge:` block returns the clear no-provider error.
