## Why

`a003` closed the credential **key-flow**: nothing passes a key to a subprocess. But a model can still go *get* a credential that exists on the host — read another CLI's config store, `~/.ssh`, or autocoder's own config; or sniff/MITM the wrapped CLI's own authenticated connection. The wrapped CLIs each ship their own "I won't let the model look around" sandbox, but trusting the sandbox of the thing you are trying to contain is not security. The boundary has to be enforced from *outside* the CLI, by the kernel.

This change is the **reach** half: a kernel-enforced sandbox around every agentic subprocess, so a model cannot reach a credential even though one exists. It builds directly on the single `agentic_run` spawn seam — wrap that one seam and no role can opt out.

It also closes the specific cross-CLI gap the multi-CLI direction opens: with `claude`, `opencode`, and other CLIs each holding their own provider credentials, nothing today stops a model running as one CLI from reading another CLI's store. A credential read into a model's context is, conservatively, disclosed to that model's provider permanently — worst of all a *cross-provider* key landing in a different provider's context. Prevention (the model never sees it) is the only control for that; this change is that prevention.

## What Changes

**Every agentic subprocess is wrapped in an OS-level sandbox.** At the `agentic_run` spawn seam, the subprocess is launched under `systemd-run` (transient service mode, so PID 1 applies the namespace), with a `bwrap` fallback for unprivileged / non-systemd hosts. The sandbox enforces, identically for every role: a **filesystem allowlist** (the workspace — read-write for the executor, read-only for audits/agentic reviewer — plus the running role's *own* CLI config store read-only for authentication, plus the minimal runtime; everything else, including other CLIs' stores, `~/.ssh`, and autocoder's config/state, absent from the namespace); **capability drops** (`CAP_NET_RAW`, `CAP_NET_ADMIN`, `CAP_SYS_PTRACE`, `NoNewPrivileges`, no `AF_PACKET`); and a **restricted `/proc`** (no other-process `environ`/`mem`). Enforcement is external to the CLI and does not depend on the CLI's own settings.

**Network egress is deliberately NOT restricted.** Outbound filtering belongs to the host firewall, not an application daemon; there is no maintainable in-app allowlist for CDN'd API/forge hosts. The sandbox does filesystem and host isolation, not a network allowlist.

**CLI config stores get two protection layers, each a toggle.** `os_hide` (the allowlist — every CLI store *other than the running role's own* is absent; fail-closed) and `engine_deny` (the existing per-invocation tool-use denylist, extended to cover every registered CLI strategy's store including the self-store; fail-open speed bump for the one store `os_hide` cannot remove). Both default ON. `os_hide` × `engine_deny` are the primitive; the operator presets (both-on default; `os_hide` off for repos that wrap CLIs and need a nested CLI to authenticate live; both off for credential-grab testing) are documentation over the two switches. Per-repo overrides global; loosening is explicit and logged.

**Fail closed.** When no sandbox mechanism is available, agentic runs fail with a clear error unless the operator explicitly opts into unsandboxed operation (logged loudly).

## Impact

- **Affected specs:** `executor` — ADD `Every agentic subprocess runs inside an OS-level sandbox` AND `CLI config stores are protected by OS-hide and engine-deny`. `orchestrator-cli` — ADD `Sandbox credential-protection config — toggles, precedence, and relaxed-posture logging`.
- **Affected code:** `agentic_run.rs` — wrap the spawn under `systemd-run` (service mode, stdout captured via `--pipe --wait --collect`) / `bwrap`; build the per-role filesystem allowlist (workspace ro/rw by role + the role's own CLI store ro); apply capability drops and `/proc` restriction; hide every non-self CLI store under `os_hide`. The tool-use settings generation — extend the default `disallowed_read_paths` to every registered CLI strategy's config store. Config + startup — `os_hide`/`engine_deny` global and per-repo with the secure default, precedence, and relaxed-posture WARN; the no-mechanism fail-closed gate and its explicit opt-out.
- **Operator-visible behavior:** agentic runs are kernel-sandboxed; the workspace is writable (executor) or read-only (audits/reviewer); no other-CLI credential, `~/.ssh`, or autocoder config is reachable. Both toggles default on. A repo that develops CLI wrappers (this one) sets `os_hide` off and is logged. Hosts without `systemd-run` or `bwrap` fail closed unless the operator opts into unsandboxed operation. Egress is unchanged (unrestricted at the daemon layer).
- **Dependencies:** builds on the canonical `agentic_run` primitive + `CliStrategy` (a56) and the canonical tool-use sandbox. Complementary to `a003` (key-flow), which should land first so no strategy plants a key inside the now-allowlisted workspace. Independent of `a004`/`a005`. Does NOT change the canonical tool-use-sandbox scoping (the CLI permission layer); this is the OS layer beside it.
- **Acceptance:** `cargo test` passes; `cargo clippy --all-targets -- -D warnings` is clean; `openspec validate a006-agentic-subprocess-sandbox --strict` passes. Tests: a path outside the allowlist is unreadable even via a `Bash` `cat`; a read-only role cannot write the workspace; another CLI's store is absent under the default AND engine-denied when `os_hide` is off; a relaxed toggle emits a per-repo startup WARN; no available mechanism fails closed unless opted out.
