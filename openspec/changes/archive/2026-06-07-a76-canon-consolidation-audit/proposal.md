## Why

Canon accretes redundantly. A project that implements Stripe and later PayPal can end up with two requirements expressing the same invariant ("outbound payment calls retry with backoff") under different titles; features added separately can leave two requirements that are really one. The cost is readability — an operator reads ten requirements to understand one concept — and drift risk, since duplicate statements of an invariant can fall out of sync.

`a75` finds requirements that *conflict*. This change finds requirements that *overlap* and proposes consolidating them, so the canon stays compact and each invariant has one home. It completes the canon-quality pair: `canon_contradiction_audit` (conflict) and `canon_consolidation_audit` (redundancy).

Consolidation is the one audit that rewrites canon, so the risk is information loss — the general-vs-specific trap: merging "all data is stored in a relational database" (a project-wide prescription) with "make PostgreSQL available" (one feature's implementation) into "all data is stored in PostgreSQL" silently erases the general rule and makes a later MariaDB impossible. The design treats that as the central hazard: the audit only ever **writes a change** (never the canon directly), every proposal is reviewed in a PR before it lands, and a merge that proves wrong after landing is recoverable by removing the change and rebuilding canon from the archive (`Rebuild canonical specs from archive`). The human is the arbiter; the audit drafts.

## What Changes

**`canon_consolidation_audit` (orchestrator-cli).** A new registered periodic audit following the specs-writing pattern of `missing_tests_audit` / `security_bug_audit`: it invokes the wrapped agent CLI through `agentic_run` (`a56`) with an `WritePolicy::OpenSpecOnly` sandbox, identifies a redundant cluster of canonical requirements, and drafts a consolidation change under `openspec/changes/` (prefix `consolidate-`) that MODIFIES the surviving requirement into the merged general form AND REMOVES the now-redundant requirement(s), preserving every non-redundant scenario. It returns `AuditOutcome::SpecsWritten(names)` so the same iteration's queue walk carries it to a PR. It is `requires_head_change = true`, default cadence `monthly`.

**RAG-assisted overlap detection.** Same spine as `a75`: enumerate the canonical requirements, and when `a21`'s RAG is enabled retrieve the nearest requirements per requirement via `query_canonical_specs` and judge each focused bundle for redundancy; degrade to a best-effort direct read with a logged caveat when RAG is off.

**Conservative v1 scope — near-duplicate consolidation only.** The audit proposes merges only where requirements express the *same invariant* under different titles. Broad capability restructuring and speculative over-fragmentation merges are out of scope for v1 (higher judgment, easier to over-reach).

**General-vs-specific guard.** The prompt forbids merging a project-wide prescription with a feature-specific implementation (the PostgreSQL trap) and errs toward not proposing when a merge would erase a broader rule. This is design intent verified by the drift audit, not pinned by a prompt-content test.

**Reviewer-facing loss surfacing.** Because consolidation is subtractive, each drafted proposal SHALL state the before/after scenario count AND list any scenario it dropped as redundant, with the reason — so the PR reviewer sees exactly what is being consolidated and can catch silent loss. The change validates `openspec validate --strict` before commit; an invalid draft is discarded with a WARN and no commit.

**Bounded.** `max_proposals_per_run` defaults to `1` (consolidation changes are denser to review than additive ones) and is operator-configurable.

## Impact

- **Affected specs:** `orchestrator-cli` — ADDED `Canon-consolidation audit`; MODIFIED `Registered periodic audits` (adds `canon_consolidation_audit`, enumerating seven, per that requirement's "adding an audit MUST update this requirement" mandate; built on a75's six).
- **Affected code:** a new audit in the `AuditRegistry` reusing the specs-writing machinery (`agentic_run`, OpenSpecOnly sandbox, post-hoc path check, `validate --strict` before commit, `SpecsWritten`), sharing the enumerate → `query_canonical_specs` neighbors → focused-judgment driver with `a75` (overlap vs conflict); embedded prompt `prompts/canon-consolidation-audit.md`; `validate_audit_type_names` known-slug list, README audit table, and `config.example.yaml` updated for spec/validator/docs alignment.
- **Operator-visible behavior:** none unless enabled (defaults `disabled`). When enabled, a redundant cluster surfaces as a `consolidate-` change → PR; the operator reviews the merge (including the scenario-loss summary), `@<bot> revise`s or rejects, and merges when satisfied. Wizard fast-path integration is out of scope (enabled via config).
- **Dependencies:** **stacks on `a75`** — both MODIFY `Registered periodic audits`, so this builds on a75's enumeration (six → seven) and archives after it. Uses `a21` (`query_canonical_specs`) opportunistically, `a56` (`agentic_run`), the periodic-audit framework's specs-writing path, the existing `LLM-driven audits validate their generated proposals before committing` requirement, and `Rebuild canonical specs from archive` as the recoverability backstop. Structurally parallels `missing_tests_audit` / `security_bug_audit`.
- **Acceptance:** `cargo test` passes; `cargo clippy --all-targets -- -D warnings` is clean; `openspec validate a76-canon-consolidation-audit --strict` passes. Tests: the audit registers OpenSpecOnly and runs agentic; it uses `query_canonical_specs` when RAG is on and logs best-effort when off; a near-duplicate cluster yields one `consolidate-` change (MODIFY survivor + REMOVE redundant) preserving non-redundant scenarios; a general+compatible-specific pair is NOT merged; the drafted proposal carries the scenario-count delta and the dropped-as-redundant list; an invalid draft is discarded without commit; an empty result is silent; the per-run cap is honored; `validate_audit_type_names` accepts the new slug and lists the seven.
