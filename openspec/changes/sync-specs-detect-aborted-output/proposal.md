## Why

The previous change (`sync-specs-rebuild-atomicity`) closed the destructive-partial-rebuild bug by adding a post-condition check that catches openspec's "exits 0 but did not actually archive" silent-skip case. With that fix in place, failed changes are correctly rolled back to archive instead of polluting the active path. But two reporting issues remain:

**1. The failure message describes the symptom, not the cause.** A real-world rebuild against a repo whose archived deltas contain broken `MODIFIED` references produces openspec stdout like:

```
member-saved-cards MODIFIED failed for header "### Requirement: ..." - not found
Aborted. No files were changed.
```

Openspec detects the broken reference, prints "Aborted. No files were changed.", and exits 0. The post-condition check correctly flags the failure, but the failure_reason field surfaces as "post-condition failed: active path still present at /tmp/.../changes/<slug>; openspec output: ..." — the operator has to read past the symptom (active path still present) to find the cause (openspec refused to apply the delta) buried in the output excerpt. For an operator triaging 7 failures, this is an avoidable cognitive cost.

**2. The PR body text is stale.** At `polling_loop.rs:1866` the PR generated by the rebuild contains:

```
**Failed changes** (left at active path for operator inspection):
```

This text predates the atomicity fix. Rollback now restores failed changes to archive — they are NOT left at the active path. The body text directly contradicts the diff. Operators reading the PR description form a wrong mental model: they expect to see active-path additions in the file tree (and worry when they don't, or worse, manually re-add the changes thinking the rebuild was supposed to). This was caught in operator triage.

## What Changes

**Detect "Aborted." in openspec stdout as a failure signal regardless of exit code.** Openspec's exit code is unreliable for the silent-skip case — the post-condition check is the safety net, but stdout pattern matching catches the failure earlier with the actual cause. The rebuild SHALL inspect `out.stdout` for the marker string `Aborted. No files were changed.` (or `Aborted.` as a leading-line prefix) and treat its presence as a failed archive call, even when `out.status.success()` is true. When detected, the failure path is the same as a non-zero exit: rollback runs, the change is recorded as failed, the failure_reason contains the full openspec output explaining WHY.

This is a workaround for upstream openspec behavior. The right long-term fix is for openspec to exit non-zero when it aborts. Until that lands, the stdout marker is the most reliable signal we have. The check is bounded: it only flags `Aborted.` in stdout — other openspec warnings (`⚠ Why section should not exceed 1000 characters`, `Specs to update:`, etc.) do not trigger it. False-positive risk is low because `Aborted.` is openspec's deliberate signal, not a generic word it uses elsewhere.

**Surface the openspec abort message prominently in failure_reason.** When the abort marker is detected, the failure_reason starts with `openspec refused to apply: <first line of openspec output containing the actual reason>` rather than `openspec archive exited 0 but post-condition failed: ...`. The full output remains available below the headline as today (truncated by the existing `truncate_for_report` cap). The post-condition check still runs and catches the case if stdout-marker detection somehow missed it — both fire independently, marker detection just runs first and produces a clearer message.

**Update the stale PR body text to match actual behavior.** In `polling_loop.rs`, change the "left at active path for operator inspection" line to accurately describe what happened. The new text reflects that failures were rolled back to archive and points the operator at the failure_reason field for the cause. Also add the `report.rolled_back` count to the rebuild's summary line (currently shows only `processed`, `successful`, `failed`) so the operator can confirm at a glance that the rollback count matches the failure count.

## Impact

- **Affected specs:** `orchestrator-cli` — one ADDED requirement covering stdout-marker detection and the failure-reason headline format. The PR-text update is implementation-only (the existing rebuild requirement scenarios already describe the report shape generically; no spec text contradicts the new wording).
- **Affected code:**
  - `autocoder/src/cli/sync_specs.rs` — extend the rebuild loop's success-path branch (currently lines 422-470) with a pre-post-condition check on `out.stdout` for the `Aborted.` marker. If detected, branch directly into `record_failure_with_rollback` with a headline-format failure_reason. Add a helper `fn detect_openspec_abort(stdout: &str) -> Option<String>` returning the matched reason line when detected, `None` otherwise.
  - `autocoder/src/polling_loop.rs:1866` — replace the "left at active path" string with `**Failed changes** (rolled back to archive — see failure reasons for openspec output explaining each):`. Update the summary line earlier in the same function to include `; <N> rolled back` when `report.rolled_back > 0`.
  - Tests:
    - Unit test for `detect_openspec_abort`: sample stdout from a real abort case returns the matched line; sample stdout from a clean archive returns `None`; stdout containing the word "aborted" in some unrelated context (e.g. inside a quoted code block) does NOT match — the matcher requires the exact `Aborted.` sentence pattern.
    - Integration test using a new `AbortedOutputArchiveRunner` stub that exits 0 and prints `MODIFIED failed for header "..." - not found\nAborted. No files were changed.`: assert the failure_reason starts with `openspec refused to apply:`, assert rollback runs, assert the change is back in archive.
    - PR-text test: snapshot the rebuild PR body for a fixture with N successes and M failures-with-rollback, assert it contains the new wording and the rolled-back count.
- **Operator-visible behavior:** failed-rebuild PR descriptions become honest — the body matches the file tree. Failure reasons lead with the actual upstream cause instead of the symptom. No change to the rollback behavior itself.
- **Breaking:** no. Reporting-only changes. The set of changes that succeed vs fail is unchanged (the post-condition check would have caught the same cases anyway); the failure messaging gets clearer.
- **Acceptance:** `cargo test` passes (new + existing). A rebuild against a fixture archive containing a change with a broken MODIFIED reference produces a PR whose body describes the failure as "openspec refused to apply: <reason>" rather than "post-condition failed: active path still present", whose summary line includes the rolled-back count, and whose "Failed changes" section header says "rolled back to archive" rather than "left at active path."
